# LangGraph Multi-Agent MIDI Editor System

**Date:** 2025-12-04
**Status:** 🔵 Planning
**Priority:** High
**Scope:** Backend + Frontend Integration

## Overview

Implement an AI-powered multi-agent system using LangGraph to intelligently edit and correct MIDI transcriptions generated by basic-pitch, with specific focus on guitar techniques (bends, hammer-ons, pull-offs, tapping) that are poorly transcribed.

### Testing Phase Strategy
During initial testing, **expose basic-pitch parameters in UI** to allow manual tuning and data collection. This will help identify optimal settings per stem type, which will later inform autonomous parameter selection based on stem analysis.

## Problem Statement

Basic-pitch provides automatic audio-to-MIDI transcription, but:
- **Guitar techniques** (string bends, tapping, hammer-ons, pull-offs) are incorrectly transcribed
- **Manual MIDI editing** is tedious and requires domain expertise
- **LLMs cannot directly comprehend audio**, requiring intermediary analysis tools
- **Need intelligent correction** that can compare audio to MIDI and suggest fixes

## Proposed Solution

Create a **LangGraph-orchestrated multi-agent system** where:
1. User selects problematic sections in MIDI/tablature
2. User describes the issue (e.g., "this part has a string bend")
3. Agent system analyzes both audio AND MIDI to understand the discrepancy
4. Agents propose corrections based on audio analysis
5. User approves changes before application

### Key Insight
Since LLMs can't process audio, agents use **specialized audio analysis tools** (essentia, librosa, aubio) to convert audio into text/numerical representations that LLMs can reason about.

### Agent Responsibilities

#### 1. **Orchestrator Agent**
- **Role:** Workflow manager, task delegator
- **Inputs:** User description of issue + time range
- **Actions:**
  - Parse user intent
  - Coordinate other agents
  - Manage approval workflow
  - Handle state transitions

#### 2. **Audio Analysis Agent**
- **Role:** Extract audio features for LLM reasoning
- **Tools:**
  - `extract_audio_section(start_time, end_time)` → Audio segment
  - `pitch_tracking(audio)` → Pitch curve over time
  - `onset_detection(audio)` → Note start times
  - `spectral_analysis(audio)` → Frequency content
  - `detect_bends(audio)` → Pitch bend events
  - `detect_vibrato(audio)` → Vibrato characteristics
- **Output:** Text description of audio features

#### 3. **MIDI Analysis Agent**
- **Role:** Extract MIDI features for LLM reasoning
- **Tools:**
  - `extract_midi_section(start_time, end_time)` → MIDI events
  - `parse_notes(midi)` → Note list with pitch/velocity/duration
  - `detect_chords(midi)` → Chord progressions
  - `get_pitch_bends(midi)` → Existing pitch bend events
  - `time_align(audio_onsets, midi_notes)` → Alignment mapping
- **Output:** Text description of MIDI events

#### 4. **Comparison Agent**
- **Role:** Identify discrepancies between audio and MIDI
- **Inputs:** Audio features + MIDI features
- **Actions:**
  - Compare pitch curves vs MIDI notes
  - Identify missing techniques (bends, slides, etc.)
  - Detect timing misalignments
  - Highlight incorrect notes
- **Output:** Discrepancy report with severity scores

#### 5. **MIDI Editor Agent**
- **Role:** Propose and apply corrections
- **Tools:**
  - `add_note(pitch, start, duration, velocity)`
  - `modify_note(note_id, **params)`
  - `delete_note(note_id)`
  - `add_pitch_bend(time, value)`
  - `add_cc_modulation(cc_num, value)`
  - `quantize_timing(notes, grid)`
- **Output:** Proposed MIDI modifications

#### 6. **Verification Agent**
- **Role:** Validate changes before applying
- **Actions:**
  - Check MIDI validity (timing, pitch ranges)
  - Ensure no conflicting edits
  - Simulate changes and verify
- **Output:** Approval/rejection + reasoning

### 4. LangGraph Agent System

```python
# backend/app/editors/midi_agent_editor/agents/graph.py

from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolExecutor

class MIDIEditorState(TypedDict):
    """Shared state across all agents."""
    audio_path: str
    midi_path: str
    section_start: float
    section_end: float
    user_description: str
    audio_features: Optional[dict]
    midi_features: Optional[dict]
    discrepancies: Optional[List[Discrepancy]]
    proposed_changes: Optional[List[Change]]
    verification_result: Optional[dict]
    user_approval: Optional[bool]
    error: Optional[str]

# Build graph
workflow = StateGraph(MIDIEditorState)

# Add nodes
workflow.add_node("orchestrator", orchestrator_agent)
workflow.add_node("audio_analyzer", audio_analysis_agent)
workflow.add_node("midi_analyzer", midi_analysis_agent)
workflow.add_node("comparator", comparison_agent)
workflow.add_node("editor", midi_editor_agent)
workflow.add_node("verifier", verification_agent)

# Define edges
workflow.add_edge("orchestrator", "audio_analyzer")
workflow.add_edge("orchestrator", "midi_analyzer")
workflow.add_edge("audio_analyzer", "comparator")
workflow.add_edge("midi_analyzer", "comparator")
workflow.add_edge("comparator", "editor")
workflow.add_edge("editor", "verifier")
workflow.add_conditional_edges(
    "verifier",
    lambda state: "approved" if state["user_approval"] else "rejected",
    {"approved": END, "rejected": "editor"}
)

app = workflow.compile()
```

async def transcribe_audio_to_midi(
    song_id: str,
    stem_name: Optional[str] = None,

    """
    Transcribe audio to MIDI with custom parameters.
    Testing phase: accepts user-defined parameters.
    """

    # Log parameters for analysis
    log_transcription_params(song_id, stem_name, config)

    return {
        "midi_path": result["midi_path"],
        "notes_detected": result["note_count"],
        "params_used": config
    }

@router.post("/songs/{song_id}/midi/edit")
async def edit_midi_with_agent(
    song_id: str,
    section_start: float,
    section_end: float,
    issue_description: str,
    stem_name: Optional[str] = None
):
    """Trigger agent-based MIDI editing workflow."""

    # Load audio + MIDI paths
    audio_path = get_audio_path(song_id, stem_name)
    midi_path = get_midi_path(song_id, stem_name)

    # Initialize state
    initial_state = {
        "audio_path": audio_path,
        "midi_path": midi_path,
        "section_start": section_start,
        "section_end": section_end,
        "user_description": issue_description,
    }

    # Run agent workflow
    result = await agent_workflow.arun(initial_state)

    return {
        "proposed_changes": result["proposed_changes"],
        "verification": result["verification_result"],
        "requires_approval": True
    }

@router.post("/songs/{song_id}/midi/approve-changes")
async def approve_midi_changes(
    song_id: str,
    change_id: str,
    approved: bool
):
    """Approve or reject proposed MIDI changes."""
    if approved:
        apply_changes(song_id, change_id)
        return {"status": "applied"}
    else:
        return {"status": "rejected"}

## Guitar-Specific Technique Handling

### 1. String Bends
**Detection:**
- Pitch curve shows smooth upward/downward glide
- Duration > 50ms
- Pitch change < 2 semitones (typically)

**MIDI Representation:**
- Add pitch bend MIDI events (CC #128)
- Adjust note velocity if needed

### 2. Hammer-ons / Pull-offs
**Detection:**
- Two notes on same string
- Second note has NO onset (legato)
- Pitch difference = 1-5 frets

**MIDI Representation:**
- Lower velocity on second note
- Add CC for legato articulation
- Potentially merge into single note with pitch change

### 3. Tapping
**Detection:**
- High-frequency onsets
- Large pitch jumps
- Percussive attack

**MIDI Representation:**
- High velocity notes
- Staccato duration
- Specific CC for tapping articulation

### 4. Slides
**Detection:**
- Pitch curve shows continuous glide
- Duration > 100ms
- Larger interval (> 2 semitones)

**MIDI Representation:**
- Portamento MIDI event
- Pitch bend sequence

## Example Workflow

### User Interaction:
```
1. User selects time range 1:23.5 - 1:27.0 on "guitar" stem MIDI
2. User writes: "This section has a string bend on the 3rd note, not two separate notes"
3. System triggers agent workflow
4. Audio Analysis Agent extracts section, detects pitch curve with bend
5. MIDI Analysis Agent sees two discrete notes (D# → E)
6. Comparison Agent identifies: "MIDI shows two notes, audio shows single note with bend"
7. MIDI Editor Agent proposes: "Merge notes, add pitch bend CC from D# to E"
8. Verification Agent checks validity, confirms safe
9. User sees proposed change with visualization
10. User approves
11. MIDI file updated, change logged
```

### Agent Reasoning (Simplified):
```
Orchestrator: "User reports bend issue in 1:23.5-1:27.0. Delegating to analyzers."

Audio Agent: "Pitch curve analysis shows:
  - Note onset at 1:24.1 (D# ~311Hz)
  - Pitch glides upward 1:24.2-1:24.5
  - Reaches E (~330Hz) at 1:24.5
  - No second onset detected
  - Conclusion: Single note with pitch bend"

MIDI Agent: "MIDI events in range:
  - Note ON: D# at 1:24.1, duration 0.3s
  - Note ON: E at 1:24.4, duration 0.6s
  - No pitch bend events
  - Conclusion: Two discrete notes"

Comparison Agent: "Discrepancy detected:
  - Audio: 1 note + bend
  - MIDI: 2 notes, no bend
  - Severity: High (incorrect transcription)
  - Recommendation: Merge notes, add pitch bend"

Editor Agent: "Proposed changes:
  1. Delete E note at 1:24.4
  2. Extend D# duration to 0.9s
  3. Add pitch bend CC: [1:24.2: 0] → [1:24.5: +2 semitones]"

Verification Agent: "Changes validated:
  - MIDI timing valid ✓
  - Pitch bend range valid (-8192 to +8191) ✓
  - No overlapping notes ✓
  - Approval recommended ✓"
```

## Risks & Challenges

### Technical Challenges
1. **Audio analysis accuracy** - Pitch tracking may fail on complex audio
2. **LLM reasoning limits** - May misinterpret audio feature descriptions
3. **MIDI limitations** - Can't perfectly represent all guitar techniques
4. **Performance** - Audio analysis can be slow for long sections

### Mitigation Strategies
1. Use multiple pitch tracking algorithms (aubio + essentia + crepe)
2. Provide rich, structured descriptions to LLMs (JSON format)
3. Document MIDI representation limitations upfront
4. Implement caching, async processing, progress indicators

### User Experience Risks
1. **Approval fatigue** - Too many confirmations
2. **Unclear changes** - Hard to visualize proposed edits
3. **Learning curve** - Complex UI

### Mitigation
1. Add "auto-approve low-risk changes" option
2. Rich visualizations (before/after piano roll)
3. Interactive tutorial, example workflows
